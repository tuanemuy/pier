# 概要 — Pier（モバイルネイティブ tmux/SSH クライアント）

最終更新: 2026-07-11

SSH先のマシンを、ターミナルエミュレータの再現ではなく**モバイルネイティブのUIで操作する** iOSアプリのプロジェクト。詳細は [USER_STORIES.md](USER_STORIES.md)（要件）と [DESIGN.md](DESIGN.md)（設計書）にまとまっている。[tmux-mobile-client-mockup.jsx](tmux-mobile-client-mockup.jsx) はUI検証用のReactモックアップ。

## 名称

**Pier（ピア）** — 桟橋。手元のデバイス（陸）とリモートマシン（海）をつなぐ接続点であり、船（セッション）を係留していつでも戻れる場所。2026-07-11決定。devツール／App Store領域に重大な競合がないことを確認済み（同名のインディーCLIエージェント等はあるが、領域・規模とも許容範囲と判断）。

## コンセプト

- コマンドはチャットのようにブロック単位で送受信する
- テキストファイルはネイティブエディタで編集する
- tmuxのセッション/ウィンドウ/ペインを構造化データとして受け取り、ネイティブUIで描画する
- 任意のTUI（vim, htop, Claude Code等）はモバイル最適化したターミナルビューにフォールバックする

## 中核となる技術判断

**tmux コントロールモード（`tmux -CC`）を接続の必須前提とする。** 接続は常に `tmux -CC new -A -s <名前>` で行い、ユーザーは常にtmuxセッションの中にいる。

- ウィンドウ/ペインの構造変更が通知として届き、描画の主導権がクライアント側にある（iTerm2が実証済みの方式）
- 切断耐性がアーキテクチャから自然に得られる。状態はサーバー側のtmuxが保持し、クライアントは再接続して復元するだけ（moshは不要）
- 任意のTUIのモバイル再構成は原理的に不可能なため、tmuxコントロールモード以外はフォールバック描画とする

iOSはアプリを確実にサスペンドするため、「接続を保つ」のではなく**「瞬時に再接続して完全復元する」**設計を取る。

## 主なUI

- **ペイン = カードデッキ**: フォーカスペインを最大化した重なり合うカードとして表示。スワイプで隣のペインへ移動し、ペインのない方向へのスワイプで新規分割する
- **チャットブロック**: OSC 133（シェル統合）でコマンドと出力を1ブロックとして表示。終了コード・実行時間・再実行・折りたたみに対応
- **ネイティブエディタ**: ファイルを開く操作でネイティブエディタが開き、保存でリモートに書き戻す。MVPはプレーンテキストの編集しやすさ（日本語IME・undoの信頼性）を優先し、シンタックスハイライトは将来対応
- **TUIフォールバック**: TUIはターミナルビューでそのまま表示。アクセサリバーから特殊キーを送信できる
- **日本語入力**: チャット入力バーでIMEがそのまま使え、TUIへも `send-keys -l` でリテラル送信する

## 技術スタック（iOSネイティブ）

| 領域 | 採用 |
|---|---|
| UI | SwiftUI（＋必要箇所UIKit） |
| SSH/SFTP | Citadel（SwiftNIO SSHベース） |
| ターミナル描画 | SwiftTerm |
| エディタ | UITextViewベースの薄い自作ラッパー（ハイライトは将来Neon等を追加） |
| tmux -CC パーサー | 自作（TmuxKit） |

モジュールは App（SwiftUI）→ TmuxKit（UI非依存の純Swiftパッケージ）→ ConnectionKit（SSH接続層）の3層構成。クロスプラットフォームは、難所がすべてiOS低レベルAPIに密着しているため採用しない。

## スコープ

MVPは23ストーリー（接続とホスト管理、セッション/ペイン操作、チャットUI、ファイル編集、TUIフォールバック、切断耐性、日本語入力、基本設定）。複数ホスト同時接続、ペインの並び替え、サーバー側エージェント（Rust製）などは意図的にスコープ外。

## 実装フェーズ

1. 接続基盤 + TmuxKit（TmuxParserのフィクスチャテストから着手）
2. カードデッキとペイン操作
3. チャットUI（OSC 133注入とブロックモデル）
4. ネイティブエディタ（SFTP + UITextViewベースのプレーンテキスト編集）
5. 復元と仕上げ
