# 設計書 — Pier（モバイルネイティブ tmux/SSH クライアント）

最終更新: 2026-07-11
関連ファイル: `USER_STORIES.md`（要件）、`tmux-mobile-client-mockup.jsx`（UIモックアップ、React製・検証用）

## 1. コンセプト

SSH先のマシンを、ターミナルエミュレータの再現ではなく**モバイルネイティブのUIで操作する** iOSアプリ。

- コマンドはチャットのようにブロック単位で送受信する
- テキストファイルはネイティブエディタで編集する
- tmuxのセッション/ウィンドウ/ペインを構造化データとして受け取り、ネイティブUIで描画する
- 任意のTUI（vim, htop, Claude Code等）はモバイル最適化したターミナルビューにフォールバックする

### 中核となる技術判断

**tmux コントロールモード（`tmux -CC`）を接続の必須前提とする。**
接続は常に `tmux -CC new -A -s <名前>`（またはセッション選択後のattach）で行い、ユーザーは常にtmuxセッションの中にいる。これにより:

- ウィンドウ/ペインの構造・変更が通知（`%window-add`, `%layout-change` 等）として届き、描画の主導権がクライアント側にある（iTerm2がデスクトップで実証済みの方式）
- 切断耐性（F系）がアーキテクチャから自然に得られる — 状態はサーバー側のtmuxが保持し、クライアントは再接続して復元するだけ
- moshは不要

任意のTUIをモバイル用に再構成することは原理的に不可能（リモートプログラムが文字グリッドを直接描画するため）。tmuxコントロールモードだけが構造化の例外であり、それ以外はフォールバック描画とする。

## 2. 技術スタック（iOSネイティブ）

| 領域 | 採用 | 備考 |
|---|---|---|
| UI | SwiftUI（＋必要箇所UIKit） | キーボードアクセサリ・テキスト入力周りはUIKit併用 |
| SSH/SFTP | Citadel（SwiftNIO SSHベース） | 鍵はSecure Enclave / Keychainに保存 |
| ターミナル描画 | SwiftTerm | TUIフォールバック用 |
| エディタ | UITextViewベースの薄い自作ラッパー | MVPはプレーン編集特化。ハイライト（D5）は将来Neon等を追加 |
| tmux -CC パーサー | 自作（TmuxKit） | 行ベースのテキストプロトコル |
| 将来: サーバーエージェント | Rust静的バイナリ（musl, aarch64/x86_64） | 高速ファイルAPI等。MVP外 |

クロスプラットフォーム（Flutter/RN）を採らない理由: 難所（ターミナル描画・キーボード制御・エディタ）が全てiOS低レベルAPIに密着しており、ブリッジのコストが本体開発を上回るため。

## 3. モジュール構成（3層）

この3層はヘキサゴナル構成として `ARCHITECTURE.md` で精緻化した（依存方向は ConnectionKit → TmuxKit に反転し、ポートはコア側が定義する）。以下は概念図として残す。

```
App（SwiftUI）
 │  カードデッキ / タブ / サイドバー / チャットUI / エディタ
 ▼
TmuxKit（純Swiftパッケージ、UI非依存）
 │  ├ TmuxParser   : 1行 → TmuxMessage enum の純関数。副作用ゼロ、テストを厚く
 │  ├ TmuxGateway  : actor。コマンド送信キュー（FIFOで %begin/%end 応答と突き合わせ）
 │  │                と非同期通知（%output, %window-add, %layout-change…）のディスパッチ
 │  └ SessionModel : 通知を畳み込んだ観測可能な状態
 │                   （セッション→ウィンドウ→ペインのツリー + ペインのグリッド座標）
 ▼
ConnectionKit
    Citadelを包んだSSH接続層。接続 → tmux -CC 起動 → バイトストリームをTmuxKitへ
```

- ペインごとの `%output`（8進エスケープ）をデコードしてSwiftTermの `feed()` へ。入力は `send-keys` で返す
- TmuxParserは実サーバーのトランスクリプトをフィクスチャにした単体テストで開発する（実機・実サーバー不要で大半を検証可能）

## 4. UI仕様

### 4.1 画面構成

- **ヘッダ**: セッション名・ウィンドウ名、接続状態（ホスト名 + tmux -CC 接続中）
- **タブ = tmuxウィンドウ**: `index:name` 表記。`+` で新規ウィンドウ
- **サイドバー（左ドロワー）**: セッション→ウィンドウのツリー。ウィンドウ数バッジ、新規セッション作成
- **アクセサリバー**: esc / tab / ctrl / C-b / 矢印（横スクロール、将来カスタマイズ可 = I4）
- **入力バー（2モード）**
  - チャットモード: `❯` プレフィックス付き入力欄 + 送信。日本語IME可（G1）
  - TUIモード: キー直接送信。ただしIMEコンポーズ用フィールドとしても機能し、確定文字列を `send-keys -l` で送る（G2）
- **トースト**: 実行したtmuxコマンドのフィードバック表示（上部）

### 4.2 ペイン = カードデッキ（確定仕様）

ウィンドウ内のペインは**重なり合うカード**。フォーカスペインを最大化し、隣接ペインはカードの縁だけを下から覗かせる。

- フォーカスカードが最前面（強い影）。ペインが1枚でも常にカードとして表示（角丸・影・ヘッダ付き）
- 隣接カードはフォーカスカードの**下に潜り**、縁だけが覗く。交差軸方向に狭め、角丸が見えることで「下に別のカードがある」ことを伝える
- ジェスチャー
  - スワイプ（上下左右）: 隣のペインへ移動（`select-pane`）。指に追従、閾値で確定/スナップバック
  - ペインのない方向へスワイプ: その方向に新規分割（`split-window -h/-v`、上・左は `-b`）
  - 覗きカードのタップ: そのペインを前面へ
  - 長押し: コンテキストメニュー（閉じる=`kill-pane` / 移動=`break-pane`,`join-pane`）（B6）
- グリッドモデル: ペインは (x, y) 座標を持つ。横分割 = x軸、縦分割 = y軸。斜め位置のカードは畳んでおき、移動すると現れる

**ジオメトリ定数（モックで調整済みの初期値）**

| 定数 | 値 | 意味 |
|---|---|---|
| M | 8 | デッキ外周マージン |
| PEEK | 16 | 隣カードが覗く量（1枚目） |
| EXTRA | 8 | 2枚目以降が追加で覗く量 |
| UNDER | 12 | 隣カードがフォーカスカードの下に潜る量 |
| CROSS | 26 | 覗きカードを交差軸方向に狭める量 |
| STEP | 6 | 奥のカードほどさらに狭める量 |
| 角丸 | 18 | カード共通 |
| ヘッダ高 | 40 | ペインID + 実行コマンド名 |

実装上の注意（モックのデバッグで得た知見）:

- ジェスチャーは必ず専用のレコグナイザで奪う（Web版では touch-action: none + Pointer Events が必要だった。iOSではUIGestureRecognizerで同種の競合設計が必要 — 特にペイン内スクロールとの排他）
- 覗きカードはフォーカスカードの交差軸から**絶対にはみ出させない**（はみ出すと前後関係が破綻して見える）
- 非フォーカスカードは描画更新を停止し（バッファのみ更新）、前面化した瞬間に再描画してGPU負荷を抑える

### 4.3 チャットブロック（シェルペイン）

- OSC 133（セマンティックプロンプトマーク）でコマンド/出力の境界を検出。接続時にシェル統合スクリプトを自動注入
- 1コマンド = 1ブロック: 左端に終了コード色のスパイン（緑=0 / 赤=非0）、コマンド行（`❯` + 太字mono）、実行時間、終了コードチップ、再実行ボタン
- 長い出力は折りたたみ（「N行の出力 — タップで展開」）

## 5. 技術ノートと既知の課題

- **ペイン幅の同期**: 常にフォーカスペインのみ全画面描画するため、tmux側には `resize-pane -Z`（ズーム）でフォーカスペインを最大化し、テキスト折り返しを画面幅と一致させる。**ズームはウィンドウ単位の状態**なので、デスクトップから同時アタッチすると表示に影響する（B7の課題）。MVPは「モバイル専用セッション」前提で割り切り、将来はクライアント側グリッドとレイアウトツリーの緩い同期を検討
- **アタッチ時の画面復元（A5/F1の基盤）**: tmux -CCはアタッチ時に既存ペインの画面内容を流さない。`capture-pane -e -p`（エスケープシーケンス付き）で取得しSwiftTermに投入する
- **iOSバックグラウンド制約**: アプリは数十秒〜数分で確実にサスペンドされSSHは切れる。「接続を保つ」のではなく**「瞬時に再接続して完全復元する」**設計とする（tmuxが状態を保持しているため成立する）
- **日本語**: G1はネイティブ入力欄なので問題なし。G2は `send-keys -l` によるリテラル送信（インライン変換表示は不可の制約付き）。I2で全角（2幅）グリフの描画検証を必須とする
- **エディタ方針（2026-07 ライブラリ調査を反映）**: MVPはハイライトより「プレーンテキストの編集しやすさ」（IME・キーボード・スクロール・undoの信頼性）を優先し、システムUITextViewベースの薄いラッパーで実装する。日本語IMEの実績はシステム標準が最も確実で、外部依存リスクもない。エディタは `RemoteFileEditor` プロトコルの裏に置いて差し替え可能にする。ハイライト導入（D5）時の候補: (a) UITextView + Neon（`TextViewHighlighter` がUITextViewを直接サポートし、エディタ乗り換え不要）、(b) STTextView + Neon、(c) Runestone。Runestoneを既定にしない理由: 2024年中頃から実質メンテ凍結（以後はXcode互換パッチのみ）で、日本語多段変換のmarkedRange未クリア（simonbs/Runestone#312）とiOS 26.1での改行挿入クラッシュ（#413）が未修正のため
- **プロトコル参照**: tmux -CC の応答は `%begin`/`%end`/`%error` 囲み・送信順FIFO。通知は `%output %<pane-id> <8進エスケープデータ>`, `%window-add`, `%window-close`, `%window-renamed`, `%layout-change`, `%session-changed`, `%sessions-changed`, `%exit` 等
